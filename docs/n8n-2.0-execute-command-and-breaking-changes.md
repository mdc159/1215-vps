# n8n 2.0: Execute Command Node & Breaking Changes

_Compiled April 20, 2026. Applies to self-hosted n8n — Docker, npm/systemd, and VPS deployments._

## TL;DR

- **Execute Command node is disabled by default in n8n 2.0**, not removed. Re-enable by clearing or editing `NODES_EXCLUDE`.
- **The `N8N_NODES_INCLUDE` workaround does not work** — there's an open bug. Use `NODES_EXCLUDE` instead.
- The 2.0 upgrade tightens many defaults (task runners, env access in Code nodes, file access, OAuth callbacks, config file permissions). Most self-hosted workflows need small fixes, not a rewrite.
- A cluster of critical CVEs (Ni8mare, N8scape, CVE-2026-27577) in late 2025 / early 2026 is what drove these hardening defaults.

## Execute Command Node

### What happened

In n8n 2.0:

- The `ExecuteCommand` and `LocalFileTrigger` nodes are on the default block list because they allow arbitrary shell commands and filesystem access.
- Existing workflows that used the node throw: `Problem running workflow — Unrecognized node type: n8n-nodes-base.executeCommand`.
- The node itself is still in the codebase — it's just excluded by default.

This change is direct fallout from the **Ni8mare CVE chain (CVE-2026-21858, CVSS 10.0)** from January 2026, where a forged admin session could be chained into a new workflow containing an Execute Command node to achieve RCE.

### Workaround — re-enable the node

Set `NODES_EXCLUDE` in your environment. It's an **exclusion** list, so the fix is to either clear it or remove only `executeCommand` from it.

**Docker Compose:**
```yaml
services:
  n8n:
    image: docker.n8n.io/n8nio/n8n:latest
    environment:
      - NODES_EXCLUDE=[]
      # Or, to disable only LocalFileTrigger but keep Execute Command:
      # - NODES_EXCLUDE=["n8n-nodes-base.localFileTrigger"]
```

**npm / systemd install (e.g. `/opt/n8n.env`):**
```
NODES_EXCLUDE=[]
```
Then:
```
sudo systemctl restart n8n
```

**Bug to avoid**: `N8N_NODES_INCLUDE=n8n-nodes-base.executeCommand` does **not** re-enable the node as of n8n 2.0. Don't burn time on it — tracked as an open GitHub issue. Use `NODES_EXCLUDE` only.

### Security caveats before flipping this

Enabling Execute Command means anyone with workflow edit access can run arbitrary shell commands on the host. Fine for a single-user homelab or isolated VPS; dangerous in any multi-user or internet-exposed instance.

If you do enable it:

- Lock down workflow edit permissions to trusted users only.
- Put n8n behind auth (reverse proxy, Cloudflare Access, or n8n's own).
- Don't expose n8n's UI publicly without MFA.
- Consider running n8n in a sandboxed container with minimal filesystem access.

### Runtime errors (still apply when enabled)

| Error | Cause | Fix |
|---|---|---|
| `command not found` | Shell can't find the binary | Check PATH of the n8n user; in Docker, verify the command exists in the container (extend the image if needed) |
| `stdout maxBuffer exceeded` | Command produces more output than the node buffers | Use flags to limit/filter output (`head`, `--quiet`, pagination) |
| Hangs forever | Command is waiting on stdin | Use non-interactive flags: `apt-get install -y`, `mysql --password=...`, etc. |
| Permission denied | Volume mounted read-only, or user mismatch | Remove `:ro`, align container `user:` to host UID, check SELinux/AppArmor |

Default shells: Linux `/bin/sh`, macOS `/bin/zsh`, Windows `cmd.exe`. In Docker, it's the container's shell, not the host's.

### Alternatives if you don't want to re-enable it

- **HTTP Request node → small wrapper service**: Flask / Express / FastAPI endpoint with its own auth, called from n8n. Keeps blast radius contained.
- **SSH node against localhost**: separate user, key-based auth, better isolation than raw Execute Command.
- **Purpose-built community nodes**: file ops, rsync, etc. — safer than generic shell.
- **Code node with `require()`** (if that's the real need): switch to external task runners and set `NODE_FUNCTION_ALLOW_EXTERNAL` to whitelist packages.

## Other 2.0 Breaking Changes Worth Knowing

### Code node: env variables blocked

`process.env.FOO` no longer works in Code nodes by default. Options:

- Move secrets to n8n credentials (preferred).
- Set `N8N_BLOCK_ENV_ACCESS_IN_NODE=false` if you really need env access.

### Task runners enabled by default

All Code node executions run on task runners now.

- **Internal mode** (default, simplest): set `N8N_RUNNERS_ENABLED=true`. No extra containers.
- **External mode** (required for Python Code nodes): run task runners as a separate container. Not bundled in the main Docker image in 2.0.

Quick decision table:

| Your setup | Mode |
|---|---|
| JS Code nodes only, no `require()` | Internal |
| Python Code nodes (anything) | External |
| Need `require()` for npm packages | External + `NODE_FUNCTION_ALLOW_EXTERNAL` |
| Maximum security / multi-tenant | External |

### File access restricted

`ReadWriteFile` and `ReadBinaryFiles` can only access `~/.n8n-files` by default. Override with `N8N_RESTRICT_FILE_ACCESS_TO` if your workflows touch other paths.

### OAuth callback requires auth

`N8N_SKIP_AUTH_ON_OAUTH_CALLBACK` default flipped from `true` to `false`. Set to `false` on 1.x before upgrading and re-verify OAuth integrations before the jump.

### Config file permissions enforced

Config files must be `0600`. On Linux/Mac:
```
chmod 600 ~/.n8n/config
```
Or disable the check with `N8N_ENFORCE_SETTINGS_FILE_PERMISSIONS=false` (needed on Windows).

### Start node removed

- Manual executions → replace with **Manual Trigger** node.
- Sub-workflows → replace with **Execute Workflow Trigger** node, then activate.
- Disabled Start nodes → delete them.

### Sub-workflow Wait behavior fixed

Parent workflows now correctly receive the end-of-workflow output from sub-executions that contained Wait (>65s), webhook, form, or human-in-the-loop nodes. Good fix, but review any logic that was compensating for the old broken behavior.

### Database changes

- **MySQL / MariaDB removed** as n8n's internal DB — PostgreSQL or SQLite only. (The MySQL *node* for querying external DBs still works.)
- **SQLite non-pooling driver removed** — pooling driver is default, WAL mode, up to 10× faster. Default pool size is 2.

### Binary data memory mode removed

Filesystem or S3 only. Budget disk headroom if you process large files.

### `--tunnel` CLI option gone

Use Cloudflare Tunnel, ngrok, or a reverse proxy instead.

### Activate/Deactivate → Save/Publish

Saving no longer pushes live. You now click **Publish** to make changes live. Autosave shipped in v2.4.0 (January 2026).

## Recommended Upgrade Path

1. **Get to 1.121.0+** (if not already) to access the Migration Report.
2. **Run Settings → Migration Report** (global admin only). Fix criticals first.
3. **Back up**: `.n8n` directory, database, workflows, credentials.
4. **Fix workflow-level issues**:
   - Replace Start nodes (Manual Trigger / Execute Workflow Trigger).
   - Replace `process.env.*` in Code nodes with credentials.
   - Decide: re-enable Execute Command via `NODES_EXCLUDE`, or replace with HTTP wrapper / SSH node.
5. **Pre-set new-default env vars on 1.x** so you can test before the jump:
   ```
   N8N_RUNNERS_ENABLED=true
   N8N_SKIP_AUTH_ON_OAUTH_CALLBACK=false
   N8N_ENFORCE_SETTINGS_FILE_PERMISSIONS=true
   ```
6. **Upgrade image / package** to latest 2.x.
7. **Post-upgrade**: re-verify every integration credential manually, retest OAuth flows, spot-check sub-workflows that use Wait nodes.

The 1.x branch (currently 1.123.25) gets roughly 3 months of security patches after 2.0, no new features. Staying on 1.x is viable short-term, not long-term — especially given the CVE density above.

## Quick Reference: Env Vars for Self-Hosted 2.0

```bash
# Re-enable Execute Command (trusted environments only)
NODES_EXCLUDE=[]

# Task runners
N8N_RUNNERS_ENABLED=true
# N8N_RUNNERS_MODE=external     # only if using Python Code nodes

# Security defaults (match 2.0 expectations, pre-set on 1.x for testing)
N8N_SKIP_AUTH_ON_OAUTH_CALLBACK=false
N8N_ENFORCE_SETTINGS_FILE_PERMISSIONS=true
N8N_BLOCK_ENV_ACCESS_IN_NODE=true
# N8N_BLOCK_ENV_ACCESS_IN_NODE=false   # only if Code nodes need process.env

# File access (widen the allowed path if needed)
# N8N_RESTRICT_FILE_ACCESS_TO=/home/node/.n8n-files,/data/ingest

# SQLite pooling (default in 2.0, safe to set earlier)
DB_SQLITE_POOL_SIZE=2
```
