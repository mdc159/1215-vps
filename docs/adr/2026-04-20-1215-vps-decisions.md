# 1215-VPS Architecture Decision Log

**Started:** 2026-04-20

Running log of consequential decisions made during the design of the
1215 VPS. Each entry uses Michael Nygard's format (Context / Decision /
Consequences / Revisit when), kept short. New decisions go at the
bottom; existing decisions are not rewritten unless superseded — if we
change our minds, we add a new entry that references the old one.

The companion canonical spec lives at
`docs/superpowers/specs/2026-04-20-1215-vps-design.md`. This file
explains **why**; the spec describes **what**.

---

## ADR-001 — VPS hosts the Orchestrator slice only

**Context.** The three-host architecture in `docs/Self-Hosted
Long-Horizon Memory Architecture…md` specifies three CEO profiles:
Engineering (WSL2), Research (macOS), Orchestrator (VPS). An all-in-one
single-host deployment is technically possible but collapses the
isolation the architecture is built around.

**Decision.** The VPS runs only the Orchestrator CEO and the shared
broker layer. Engineering and Research live on other hosts and publish
into the broker over Tailscale.

**Consequences.** Simpler VPS surface (one profile instead of three,
one set of Hermes state). Other two hosts are out of scope for this
repo until their own specs exist.

**Revisit when.** We ever want to collapse due to not having the other
two hosts available — at which point we accept the isolation tradeoff
explicitly.

## ADR-002 — Compose overlay, not patched submodules

**Context.** `local-ai-packaged` is a submodule; so are Honcho, Paperclip,
etc. We need to add 1215-specific services (Honcho, broker-db,
paperclip-orchestrator, n8n-mcp, init jobs) alongside them. Options
considered: edit `local-ai-packaged/docker-compose.yml` in place;
write a clean top-level overlay; reimplement everything.

**Decision.** Keep all submodules unedited. Write
`stack/docker-compose.1215.yml` as an overlay layered via `-f` on top
of upstream composes, sharing project name `localai` and networks.

**Consequences.** Upstream pulls stay clean. One extra file to
maintain, plus `bin/start-1215.py` that knows the stacking order.
Overlay can only add services; it never redefines upstream ones.

**Revisit when.** Upstream `local-ai-packaged` diverges so far from
what we need that layering becomes uglier than forking. Low probability.

## ADR-003 — Hermes runs on host; containers reach it via Unix-socket gateway

**Context.** The `hermes-paperclip-adapter` spawns `hermes` as a
subprocess. User intends to use Hermes for many things beyond
Paperclip (standalone VPS tasks, self-evolution experiments, future
Paperclip sub-roles). Three packaging options were considered:
(a) single image bundling Paperclip + Hermes, (b) sidecar containers,
(c) Hermes on host, Paperclip in container.

**Decision.** Option (c). Hermes binary and per-profile state
(`/var/lib/hermes/profiles/*`) live on the host. A small
`hermes-gateway` daemon (systemd unit) listens on
`/run/hermes-gateway/hermes.sock`. Containers that need Hermes
bind-mount the socket and ship a `hermes` shim in their image that
forwards calls to the socket; the adapter's subprocess model is
preserved unmodified.

**Consequences.** Maximum flexibility — any client (Paperclip
container, n8n, MCP servers, ad-hoc CLI on host) can hit the same
Hermes through the same gateway. Profiles are just directories; adding
profiles requires no config change. Downside: one more host component
to install and monitor (the gateway), and container images need a
10-line shim.

**Revisit when.** The adapter gains first-class RPC/MCP support, at
which point we can drop the shim and point clients directly at the
gateway's HTTP port.

## ADR-004 — Gateway exposes Unix socket, not SSH, not MCP-first

**Context.** Given ADR-003, containers need to reach the host Hermes.
Options: Unix socket on a bind mount (clean filesystem semantics);
SSH shim (simple but adds SSH handshake per call); MCP server in
front of Hermes (future-facing but requires adapter changes today).

**Decision.** Unix socket. Paperclip container bind-mounts
`/run/hermes-gateway/hermes.sock`; a shim binary in the image forwards
argv + stdin over the socket.

**Consequences.** Low-latency, no network hop, no auth complexity
inside the host. Adapter stays unmodified. An MCP server can be added
*on top of* the gateway later without changing anything underneath.

**Revisit when.** MCP support becomes the canonical way agents call
each other, or when we need network-transparent access from off-host
(at which point we add an authenticated HTTP frontend to the same
gateway).

## ADR-005 — Data layer: Honcho reuses Supabase; broker gets its own Postgres; Langfuse stays as upstream ships it

> **SUPERSEDED by ADR-016 (2026-04-20).** The broker-in-its-own-Postgres
> half of this decision was wrong: remote CEOs on other hosts need an
> ingress path, and a hidden second Postgres had none. ADR-016 moves
> the broker into a schema inside Supabase so remote CEOs can reach it
> via Supabase's existing PostgREST on a tailnet vhost. The Honcho and
> Langfuse parts of this decision still stand.

**Context.** Three potential Postgres consumers (Honcho, alignment
log, Langfuse). Consolidation options ranged from "everything into
Supabase" to "a Postgres per consumer."

**Decision (superseded).** Split by blast radius:
- **Honcho** reuses Supabase `db` with a dedicated `honcho` database
  and `honcho_app` user (per `docs/honcho.md`). **Still in force.**
- **Alignment log + artifact manifests** live in a new, dedicated
  `broker-db` (pgvector pg15) in its own postmaster. **Superseded.**
- **Langfuse** keeps its upstream-bundled Postgres — tightly coupled,
  not worth fighting. **Still in force.**

**Why superseded.** The original architecture doc (`docs/Self-Hosted
Long-Horizon Memory Architecture…md`) explicitly chose a
**Supabase-backed** alignment log. A separate `broker-db` instance has
no existing HTTP/auth surface, so remote CEOs would have needed either
raw Postgres-over-Tailscale (ugly, hard to RLS) or a new broker API
service (extra surface to build and secure). Moving the broker into a
schema inside Supabase's existing `db` gets PostgREST + GoTrue + RLS
for free and restores the architecture doc's intent.

## ADR-006 — Self-host Supabase (not cloud)

**Context.** Cloud Supabase was considered for operational ergonomics
(managed backups, upgrades, pooler, dashboard).

**Decision.** Self-host on the VPS.

**Consequences.** Every query crosses Docker-internal networking, not
the public internet — lower latency for Honcho/broker workloads that
hammer the DB with small queries. All memory and alignment-log data
stays on the VPS (matches `docs/honcho.md` "do not expose publicly"
stance). Ops burden is higher; we rely on `restic` and `pg_dump`
rather than managed backups.

**Revisit when.** Ops burden of self-hosted Supabase becomes painful
(version upgrades, failed migrations), or we need features only the
managed service offers.

## ADR-007 — MinIO is the unified artifact and binary-data surface

**Context.** Upstream MinIO is already in the stack (used by
Langfuse). n8n, by default, stores binary workflow output in
Postgres/filesystem — awkward to retrieve from a Windows desktop.
Paperclip also needs artifact storage.

**Decision.** Point n8n at MinIO for binary data
(`N8N_DEFAULT_BINARY_DATA_MODE=s3`). Create a second bucket
`artifacts` for Paperclip outputs. The existing `langfuse` bucket
stays. User retrieves files from any box on the tailnet via rclone or
an S3 browser against `minio.tail-...ts.net`.

**Consequences.** One S3-compatible surface for all "stuff we want to
pull off the VPS." Single auth (MinIO root creds). MinIO is on the
critical path for n8n — an outage blocks workflows that write files.

**Revisit when.** Artifact volume grows enough to justify offloading
MinIO to a dedicated box, or we want per-user MinIO IAM instead of
shared root creds.

## ADR-008 — Edge: Tailscale for admin, Cloudflare Access for exactly two public apps

**Context.** Need to access services from Windows/Mac/WSL2 (always
trusted) and occasionally from a public PC (library, travel laptop —
no client install possible). `docs/honcho.md` explicitly forbids
Tailscale Funnel.

**Decision.**
- Every admin surface (Supabase Studio, Langfuse, Neo4j, MinIO
  console, Qdrant, Flowise, SearXNG, Ollama if enabled) is served via
  Caddy on tailnet-only vhosts using Tailscale cert API. Not reachable
  without a Tailscale client.
- Exactly two apps are public: Open WebUI and n8n. They sit behind
  Cloudflare Tunnel → Cloudflare Access (email-OTP policy) → Caddy.
  Each app's native auth is the second factor.
- Cloudflared is a host systemd unit; it connects outbound to
  Cloudflare, so the VPS's public 443 only serves CF-Access-validated
  traffic plus webhook endpoints.

**Consequences.** Public attack surface shrinks to two apps. Library-PC
use case is supported. No self-hosted auth proxy (Authelia, Authentik)
needed. Adds a Cloudflare dependency for the public path.

**Revisit when.** We want webhook endpoints on many services (at which
point we add more CF Access policies, not more apps public-by-default),
or we move off Cloudflare.

## ADR-009 — Authentication via Cloudflare Access, not Supabase Auth

**Context.** Considered using Supabase Auth (already in the stack) as
the SSO layer for public apps.

**Decision.** Cloudflare Access email-OTP for the two public apps.
Supabase Auth is reserved for future application-layer use (a
Paperclip-facing UI we might build), not for gating reverse-proxied
third-party apps.

**Consequences.** Zero custom auth code. Email-OTP works on any device
with no Tailscale / no VPN / no admin rights. Supabase Auth remains
available for app work without being entangled with infrastructure
access.

**Revisit when.** We build a first-party UI that should share users
with infrastructure access, at which point we may front-door it behind
the same CF Access policy and keep Supabase Auth as the application
user DB.

## ADR-010 — Single root `.env`, with rendering step for Hermes profiles

**Context.** Every service (upstream + new) reads env vars. Per-service
`.env` files are the source of "stuff drifted in unnoticed" pain the
user has hit before.

**Decision.** One `stack/env/.env` at the repo root. Every Docker
Compose service reads it (directly or via Compose `env_file:`). Every
systemd unit reads it via `EnvironmentFile=`. Supabase's compose
requires a copy at `supabase/docker/.env` — upstream `start_services.py`
already handles that. Hermes profiles require `.env` in their profile
dir — `bin/start-1215.py` renders a curated subset on every bring-up.

**Consequences.** Single source of truth for secrets. Renders regenerate
on every start, so profile dirs can't drift from root. `.env.example`
is the canonical index (~40–60 keys, each commented with consumer).

**Revisit when.** We add a secrets manager (Vault, SOPS, Infisical) —
the root `.env` becomes its render target instead of the hand-edited
source.

## ADR-011 — `start-1215.py` generates missing secrets on first run

**Context.** Upstream `local-ai-packaged` generates the SearXNG secret
if missing; every other secret is copy-paste-from-README. That's how
people ship weak passwords.

**Decision.** `start-1215.py` detects every blank/missing required
secret in `.env` and writes a freshly-generated value in-place.
Applies to Postgres passwords, JWT secrets, Neo4j auth, ClickHouse
password, MinIO root, Langfuse secrets, n8n encryption keys, Honcho
and broker DB passwords, Flowise credentials, etc.

**Consequences.** Fresh installs have cryptographically random secrets
by default. User only edits `.env` to set things that can't be
generated: domain names, Cloudflare tunnel token, Tailscale authkey,
LLM API keys, Langfuse init org/project metadata.

**Revisit when.** We move to a proper secrets manager (see ADR-010
revisit).

## ADR-012 — autoreason and hermes-agent-self-evolution are libraries, not services

**Context.** Both submodules contain research code (experiments,
evaluations, reports), not long-running daemons.

**Decision.** Clone them onto the VPS filesystem under `modules/`.
No compose service definitions. Invoked on demand with UV Python
environments. They may call into the running stack via the same
interfaces a user would (gateway socket, Honcho API, Langfuse SDK).

**Consequences.** Running stack stays lean. Experimental runs use the
same shared services as production, so findings transfer directly.
Downside: each run is a manual invocation; no dashboard. Acceptable
for the "I'm tuning an agent" use case.

**Revisit when.** Either project adds a persistent daemon mode worth
running continuously, or we want scheduled evolution runs (cron service).

## ADR-013 — Langfuse via env-var auto-instrument day 1; LiteLLM reserved for later

**Context.** Need LLM-call observability from day 1. Two paths:
env-var SDK auto-instrument (cheap, covers SDK-using code only);
LiteLLM proxy in front of all calls (universal, requires extra service).

**Decision.** Start with env-var (`LANGFUSE_HOST`, `LANGFUSE_PUBLIC_KEY`,
`LANGFUSE_SECRET_KEY`) pre-wired into Hermes profile `.env` renderings
and the Paperclip container. Reserve a port slot for LiteLLM
(`litellm:4000`) in the port map; don't deploy it. If tools start
slipping through uninstrumented, drop LiteLLM in and flip one base-URL
env var.

**Consequences.** Zero-config observability for anything using the
Anthropic/OpenAI/Langfuse SDKs. Blind spots for tools that bypass SDKs
are a known risk, accepted for now.

**Revisit when.** We see silent LLM calls in the wild (cost anomaly,
missing traces), or we add a new model provider without SDK-level
Langfuse support.

## ADR-014 — Split Docker networks: `localai-data` and `localai-app`

**Context.** All containers could share one network; simpler. But a
compromised app-plane service (Flowise, n8n, or an agent running
untrusted prompts) would be able to port-scan the data plane directly.

**Decision.** Two Docker networks. Data-plane services (Postgres,
Valkey, Qdrant, Neo4j, ClickHouse, MinIO, broker-db) live in
`localai-data`. App-plane services (Open WebUI, n8n, Paperclip,
Honcho, Langfuse, Flowise, SearXNG) live in `localai-app`. A subset of
data-plane services (the ones apps actually call) are dual-attached.

**Consequences.** Small config overhead (dual-attach list to
maintain). Extra defense-in-depth — a flowise RCE can't probe
`broker-db` directly, only via services it's allowed to call. Worth it
given the agent workload accepts adversarial-ish prompts.

**Revisit when.** The dual-attach list becomes unwieldy, or we gain
per-service network policies via something like Cilium.

## ADR-015 — Strict halt-on-failure with three-level readiness gates

**Context.** Supabase in particular takes 2–3 minutes to finish
migrations on a cold VPS. Hardcoded sleeps (as upstream uses) race
dependent services; `depends_on: condition: service_healthy` catches
container health but not protocol-level readiness.

**Decision.** `start-1215.py` halts on first failure of any phase, and
uses three-level readiness gates for critical dependencies: container
healthy → protocol accepts queries → full-stack endpoint returns 200.
First-boot timeout for Supabase is 600s; subsequent boots 120s.

**Consequences.** No half-started states. Explicit, fast feedback on
what broke. First-boot on a cold VPS can take 5+ minutes, which is
acceptable because it's one-time.

**Revisit when.** Supabase startup becomes materially faster or slower
in future upstream versions, at which point the timeout budgets are
tuned.

## ADR-016 — Broker lives as a `broker` schema inside Supabase's `db`, not in a separate Postgres instance

**Supersedes the broker half of ADR-005.**

**Context.** External review caught a critical hole in the first draft:
an isolated `broker-db` postmaster had no ingress path for the
off-host CEOs (WSL2 Engineering, macOS Research). The `docs/Self-Hosted
Long-Horizon Memory Architecture…md` architecture doc had explicitly
specified a Supabase-backed alignment log. We drifted from that
without a reason and without paying attention to who actually writes
to the broker from elsewhere.

**Decision.** The broker is a dedicated **`broker` schema** inside the
Supabase `db` (in the existing `postgres` database), owned by a
dedicated Postgres role `broker_app`. Tables: `broker.alignment_log`,
`broker.artifact_manifests`. `db-init` creates the schema, role,
tables, and row-level policies on first boot.

- **Orchestrator (on-VPS)** writes directly via libpq on the Docker-
  internal network as `broker_app` (fast path).
- **Engineering / Research (off-host)** reach it through Supabase's
  existing `kong` → `rest` (PostgREST) service, exposed on the tailnet
  vhost `supabase.tail-…ts.net/rest/v1/broker_*`. PostgREST
  auto-exposes the schema. Auth is Supabase JWT (long-lived
  service-role JWT or GoTrue-issued user JWT); RLS policies enforce
  per-peer access.

**Consequences.**
- One fewer Postgres instance (2 postmasters instead of 3).
- Remote CEOs use the same Supabase API surface they'd use for any
  other table — no bespoke broker API to build.
- Blast radius increases: a Supabase DB outage also stalls the
  alignment log. Mitigated by Supabase being the most-battle-tested
  service in the stack and by `db` being backup-class A.
- Write performance for on-host Orchestrator is unchanged (direct
  libpq).
- Matches the original architecture doc's explicit choice.

**Revisit when.** Broker write volume or isolation requirements grow
to where co-residency with app data is a real contention issue.

## ADR-017 — n8n-mcp runs in HTTP mode, tailnet-exposed behind bearer auth

**Context.** `modules/n8n-mcp` supports stdio mode (local MCP clients
like Claude Desktop on the same machine) and HTTP mode (remote MCP
clients, multiple simultaneous consumers, bearer-token auth). Our
consumers are cross-container (Paperclip, future others) and
cross-host (Claude Code on Windows via tailnet).

**Decision.** HTTP mode. Service config:
- `MCP_MODE=http`
- `PORT=3000` inside the container, mapped to host-internal 3033 to
  avoid collision with Langfuse worker's 3030.
- `AUTH_TOKEN=${N8N_MCP_AUTH_TOKEN}` (generated by `start-1215.py`).
- `N8N_API_URL=http://n8n:5678`, `N8N_API_KEY=${N8N_API_KEY}`
  (user-provided from n8n UI after first n8n boot) — enables the
  16 n8n management tools.
- Reachable at tailnet vhost `n8n-mcp.tail-…ts.net` via Caddy.
  Clients pass `Authorization: Bearer <token>`.

**Consequences.** Single deployment serves Paperclip container (via
Docker network), host processes (via localhost), and Windows Claude
Code (via tailnet). stdio mode is not supported for this installation;
if needed for a specific local client, spin up a separate short-lived
process.

**Revisit when.** Needed by a Claude Desktop instance that can't hit
HTTP MCP, or if we want to avoid the bearer-token rotation burden via
mTLS.

## ADR-018 — Gateway protocol is a canonical interface, specified in the spec

**Context.** Finding 4 of the external review: the Paperclip → Hermes
integration seam (persistSession, --resume, extraArgs, --profile, cwd,
env merging, --yolo) is the most failure-prone part of the whole
architecture, and in the first draft it was hand-waved as "shim
forwards to gateway."

**Decision.** The spec's section 9.5 (Gateway protocol) is canonical:
- Framed-JSON preamble with `argv`, `env`, `cwd`, `tty` keys.
- Bidirectional length-framed streaming for stdin/stdout/stderr.
- Gateway is authoritative for `HERMES_HOME` resolution from
  `--profile` (protected-env mechanism prevents container override).
- `cwd` is validated against `/var/lib/hermes/workspaces/<profile>/`
  and bind-mounted at the same path on both host and container so
  paths translate symmetrically.
- Per-profile mutex in the gateway prevents the "two instances
  against the same SQLite profile" corruption mode the architecture
  doc flags. Different profiles run in parallel.
- All adapter-set flags pass through unchanged.

**Consequences.** The adapter is unmodified (matches ADR-003 intent).
The gateway becomes a small, testable piece with a written contract.
The protocol is language-independent (ADR-018 is independent of
ADR-003's Go-or-Rust choice). Testing the seam is possible without
Paperclip (section 12 tests 13 and 14).

**Revisit when.** The adapter gains first-class MCP/RPC support
(which would obsolete the shim approach entirely — see ADR-003's
revisit clause).

## ADR-019 — Honcho environment is fully specified in the secrets map, not left to the reader

**Context.** Finding 2 of the external review: the first draft named
only `HONCHO_DB_PASSWORD`, but the real Honcho self-host contract
requires a full `DB_CONNECTION_URI` (with `postgresql+psycopg://`
prefix), a `CACHE_URL`, LLM provider env for the deriver, and
optional embedding provider env. Without these, Honcho can't boot
deterministically from the spec alone.

**Decision.** The secrets map in the spec lists every Honcho key
needed by `honcho-api` and `honcho-deriver`, distinguishes generated
vs. composed vs. user-provided vs. static, and commits to `redis://
redis:6379/1?suppress=true` as the cache URL (DB index 1 to avoid
Langfuse's queue on index 0). The full `DB_CONNECTION_URI` is
composed at render time from `HONCHO_DB_PASSWORD`; users only set
the password in `.env`, not the URI.

**Consequences.** `start-1215.py`'s `.env` preflight can verify
Honcho's env completeness before attempting to boot the service.
LLM provider selection is explicit (`HONCHO_LLM_PROVIDER`,
`HONCHO_LLM_BASE_URL`), so switching Honcho to a local Ollama
endpoint is a `.env` edit, not a code change.

**Revisit when.** Honcho upstream changes its env contract (new
required variable, renamed key). Caught by reading Honcho changelogs
on submodule bumps.

## ADR-020 — Overlay introduces networks; upstream compose is never edited

**Context.** Finding 3 of the external review: the first draft's
service inventory described `localai-app` / `localai-data` networks
as if they were pre-existing in the upstream `local-ai-packaged`
compose. They aren't — upstream has no `networks:` directives at all,
and service names like `langfuse-postgres` / `valkey` were incorrect
(actual names: `postgres` / `redis`).

**Decision.** The `localai-app` / `localai-data` split is introduced
by `stack/docker-compose.1215.yml` via Compose's merge semantics. The
overlay declares the networks at the top level and re-declares each
affected upstream service with a `networks:` block listing its
memberships. This is a legal Compose merge (multi-`-f` layering) and
does not modify files in `modules/`.

**Consequences.** ADR-002's "never edit upstream" rule holds. The
spec's service inventory shows the **final** network membership after
layering, not what's declared in each source file — this is called
out explicitly at the top of the inventory section. Upstream service
names in the inventory match actual upstream (e.g., `postgres`,
`redis`, `kong`), not the mislabels the first draft used.

**Revisit when.** Upstream adopts its own network split, at which
point we align to their names rather than imposing ours.
