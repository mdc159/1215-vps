Findings
High: The shared broker layer is not actually reachable in the way the spec claims. In docs/superpowers/specs/2026-04-20-1215-vps-design.md, broker-db is defined as internal-only, while the same spec says Engineering and Research publish into it over Tailscale and even validates writes "via Supabase Studio". In the original design, docs/Self-Hosted Long-Horizon Memory Architecture for Three Hermes-Backed.md explicitly chose a Supabase-backed alignment log as the primary substrate. With the current building blocks, there is an admin/API surface for supabase-db, but no equivalent access path for a second hidden Postgres. As written, the canonical broker is unreachable from the non-VPS hosts unless you also specify a broker API, a tailnet-exposed Postgres policy, or a Supabase-facing proxy.

High: Honcho’s runtime contract is underspecified, so honcho-api and honcho-deriver are not reproducibly configurable from this spec alone. The secrets map in docs/superpowers/specs/2026-04-20-1215-vps-design.md gives you HONCHO_DB_PASSWORD, but the actual Honcho self-host contract in modules/honcho/docker-compose.yml.example and modules/honcho/README.md requires a full DB_CONNECTION_URI with the postgresql+psycopg prefix, plus concrete LLM provider keys for derivation/embeddings. Right now the spec describes Honcho as a first-class service, but not the env/rendering rules needed to boot it deterministically.

Medium: The service inventory presents the upstream local-ai-packaged module as if it already has the draft’s network and service topology, but the file it cites does not match those facts. docs/superpowers/specs/2026-04-20-1215-vps-design.md says local-ai-packaged/docker-compose.yml provides localai-app / localai-data and service names like langfuse-postgres and valkey. The actual modules/local-ai-packaged/docker-compose.yml has no explicit custom networks at all, and Langfuse currently depends on postgres and redis. That means the isolation model may be a reasonable overlay design, but it is not yet a faithful description of the current module building blocks.

Medium: The original idea treats the Hermes Paperclip adapter as a load-bearing boundary, but the draft spec leaves it implicit. docs/Self-Hosted Long-Horizon Memory Architecture for Three Hermes-Backed.md makes adapter behavior like persistSession, --resume, extraArgs --profile, cwd, env, and --yolo central to continuity and isolation, and modules/hermes-paperclip-adapter/src/server/execute.ts is where those semantics actually live. In the draft, paperclip-orchestrator is just "FROM paperclip, adds hermes-shim and adapter wiring", so the most failure-prone integration seam is not specified tightly enough to be canonical.

Open Questions
Is broker-db an intentional replacement for the original Supabase-backed alignment log, or should the spec revert to the original "Supabase table + thin broker skills" design?
If broker-db is intentional, what is the sanctioned remote ingress path for WSL/macOS: direct Postgres over Tailscale, a broker HTTP API, or something mediated by Supabase/n8n?
Is n8n-mcp supposed to run in stdio mode or HTTP mode? modules/n8n-mcp/docs/SELF_HOSTING.md and modules/n8n-mcp/docs/HTTP_DEPLOYMENT.md describe very different deployment models, and the spec currently mixes them.
Summary
The draft is strong on intent, topology, and operational discipline, but it currently overstates how closely the upstream modules already match that topology. Before treating it as canonical, I’d tighten the broker decision, make the Honcho env contract explicit, and clearly separate "current upstream behavior" from "1215 overlay we will add in stack/."
