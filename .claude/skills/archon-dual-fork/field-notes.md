2026-04-20 | REVIEW | t3-secrets-generate-hex-pure-f | [breaking] REVIEW prompt's `ID_MISMATCH` guard requires the plan file header to contain the Archon Project ID, but plan files written before the Archon project exists won't have one. First REVIEW fork halted at the guard before grading any AC. Workaround: orchestrator stamps `**Archon Project ID:** <uuid>` into the plan header and commits before dispatching REVIEW. Real fix candidates: (a) make `build_prompts.py` auto-stamp the plan, (b) soften the guard to a warning when the ID is missing entirely (vs present-but-mismatched), (c) document the requirement in the skill's "When to Use" section.
2026-04-20 | ORCHESTRATOR | t3-secrets-generate-hex-pure-f | AC-writing pitfalls (orchestrator-side defects, surfaced by reviewer's literal grading): (1) Don't constrain commit count beyond prologue using absolute SHAs — the orchestrator may make scaffolding commits between prologue and IMPL (e.g. the ID-stamp commit). Use commit-content checks (subject + changed-files set) instead. (2) Don't require global `git status` clean — the repo may have pre-existing untracked/modified state outside IMPL's scope. Constrain to "no NEW untracked files outside the listed scope" or check only the IMPL-relevant subtree. (3) Don't require tooling (ruff/mypy) the plan never adds to dev deps — IMPL ends up using ephemeral `uv run --with` workarounds that strict literal grading marks as NOT MET. ACs must reflect plan-as-written, not plan-as-wished.
2026-04-20 | ORCHESTRATOR | t3-secrets-generate-hex-pure-f | [breaking] No "AC-defect retry" path in the cookbook. The prescribed FAIL→retry flow assumes the IMPL needs to fix its code. When the failure is actually AC-side (orchestrator wrote bad criteria), there's no clean way to re-run REVIEW with corrected exam — `build_prompts.py` skips regenerating the REVIEW prompt at attempt>1 ("review prompt is stable across retries"). Workaround: delete the stale `/tmp/codex-review-{slug}-prompt.txt` to force regeneration. Real fix: add `--force-regen-review` flag, or a separate "ac-defect-retry" attempt category that resets both prompts and treats the run as a fresh attempt 1 against a new exam.
2026-04-20 | IMPL | t3-secrets-generate-hex-pure-f | [breaking] On attempt 2 with remediation context that explicitly told IMPL "verify-and-no-op expected", the Sonnet IMPL wrapper SKIPPED the Codex fork and did the verification itself in-line ("not a forked Codex subprocess — the IMPL prompt was executed directly by this agent seat per the no-op path"). Functionally fine for verify-only, but it violates the structural contract that IMPL work happens in a separate Codex fork (whole point of dual-fork). Result: no done.json, no output.log for this attempt, audit trail incomplete. Fix candidates: (a) `archon-task-implementor.md` adds explicit rule "ALWAYS fork Codex, even on verify-only — the fork is the structural guarantee, not just a workhorse choice"; (b) skill adds an explicit `mode: verify` short-circuit that documents the deviation so the orchestrator doesn't have to detect it after the fact; (c) build_prompts adds an `--expected-outcome` field that the IMPL seat must respect explicitly rather than letting the wrapper decide.
2026-04-20 | ORCHESTRATOR | t3-secrets-generate-hex-pure-f | `build_prompts.py` returns `"review_prompt_unchanged": true` whenever `attempt > 1` AND the file exists — even when the orchestrator just deleted the stale file moments before and the script wrote a fresh one. The flag is misleadingly named; it should be either `review_prompt_carried_over` (tracking whether content was reused) or computed by hashing pre/post. Minor: makes reading the script's JSON output less informative for retry diagnostics.
